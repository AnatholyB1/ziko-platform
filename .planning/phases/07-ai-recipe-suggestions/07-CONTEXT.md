# Phase 7: AI Recipe Suggestions - Context

**Gathered:** 2026-03-29
**Status:** Ready for planning

<domain>
## Phase Boundary

Users can request AI-generated recipe suggestions from a dedicated Recipes screen within the pantry plugin. Suggestions are macro-aware (respecting remaining daily calorie/protein budget) and use available pantry items. Users can view full recipe detail (ingredients, macros, cooking steps) and adjust serving count with live macro recalculation.

New capabilities NOT in scope: logging a cooked recipe to Nutrition (Phase 8), shopping list from missing ingredients (Phase 9), persisting a recipe library (future v1.2).

</domain>

<decisions>
## Implementation Decisions

### Recipe Generation Architecture

- **D-01:** Recipes are **ephemeral** — generated on-demand by AI, stored in Zustand only for the current session. No `recipes` table in Supabase (architecture decision from STATE.md v1.1). Fresh generation each time the user requests.

- **D-02:** Dedicated backend endpoint `POST /pantry/recipes/suggest` — NOT via the existing conversational AI chat endpoint. Rationale: structured output required (ingredients with quantities + macro numbers for RECIPE-04 serving adjustment), clean separation from chat tool budget (STATE.md blocker).

- **D-03:** Endpoint uses **`generateObject`** from Vercel AI SDK v6 with **Zod schema** for typed structured output. Add `zod` as backend dependency. The schema produces an array of `Recipe` objects.

- **D-04:** Endpoint internally fetches pantry items (from `pantry_items` table) AND today's nutrition summary (from `nutrition_logs`) so the AI receives full context without a round-trip. The `fetchUserContext` already has `todayNutritionSummary` — the new endpoint duplicates this fetch to remain self-contained.

- **D-05:** Endpoint input: `{ preferences?: string }` (optional user freetext e.g. "végétarien", "rapide"). Returns `{ recipes: Recipe[], remaining_macros: MacroBudget }`.

### Recipe Data Model

- **D-06:** Recipe types defined in `plugins/pantry/src/types/recipe.ts`:
  ```ts
  interface RecipeIngredient {
    name: string;
    quantity: number;
    unit: string;
  }
  interface RecipeMacros {
    calories: number;
    protein_g: number;
    carbs_g: number;
    fat_g: number;
  }
  interface Recipe {
    id: string;          // generated UUID by AI (for React key)
    name: string;
    description: string;
    prep_time_min: number;
    base_servings: number; // default serving count (usually 1 or 2)
    ingredients: RecipeIngredient[];
    macros: RecipeMacros; // per base_servings
    steps: string[];
  }
  ```

- **D-07:** Zod schema in backend endpoint mirrors the Recipe interface. The `id` field is a short string (not UUID) generated by Claude as a slug or short label.

### Serving Adjustment (RECIPE-04)

- **D-08:** Serving adjustment is **client-side only** — multiply macros by `(selectedServings / base_servings)`. No backend call needed. Multiplier applied in RecipeDetail component.

- **D-09:** Serving count selector: stepper control (− / count / +), range 1–8, integer only.

### Screens and Navigation

- **D-10:** Two new screens:
  1. `PantryRecipes.tsx` — tab-bar route (`showInTabBar: true`) — shows "Suggérer des recettes" CTA, macro budget banner, recipe cards list, optional preference input
  2. `RecipeDetail.tsx` — detail screen (`showInTabBar: false`) — full recipe, ingredients, macro table, serving adjuster, steps — navigated to from recipe card tap

- **D-11:** Recipe card tap passes recipe data via route params (JSON-encoded). RecipeDetail parses from `useLocalSearchParams`. This avoids a separate store fetch.

- **D-12:** Macro budget banner on PantryRecipes screen: shows remaining calories / protein / carbs / fat based on a 2000 kcal / 150g P / 250g C / 65g F default target minus today's logged nutrition. Display-only; no goal storage (out of scope for Phase 7).

### Pantry Recipes Backend Route

- **D-13:** New file `backend/api/src/routes/pantry-recipes.ts` with router mounted as `app.route('/pantry', pantryRecipesRouter)` in `app.ts`.

- **D-14:** Claude model for recipe generation: same `claude-sonnet-4-20250514` as the orchestrator agent. `maxTokens: 2000` sufficient for 3 recipes with full steps.

- **D-15:** System prompt instructs Claude to output 3 recipes, each using primarily items from the pantry list, respecting remaining macro budget, with realistic French-style recipe names.

### State Management

- **D-16:** Extend `plugins/pantry/src/store.ts` with recipe state:
  - `recipes: Recipe[]` — current suggestions
  - `recipesLoading: boolean`
  - `recipesError: string | null`
  - `setRecipes`, `setRecipesLoading`, `setRecipesError`

### i18n

- **D-17:** Add `pantry.recipes.*` and `pantry.recipe_detail.*` keys to central `packages/plugin-sdk/src/i18n.ts`. Follow existing naming patterns.

### Claude's Discretion

- Exact Zod schema field descriptions and prompt engineering details
- RecipeDetail layout — card stack vs scroll view for steps
- Error and empty-state copy
- Loading skeleton design for recipe cards

</decisions>

## Existing Code Insights

### Reusable from Phase 6
- `PantryItem` interface from `store.ts` — ingredients in recipes reference item names/units
- `usePantryStore` — recipes screen should prime from store (already loaded by dashboard) or re-fetch

### Vercel AI SDK v6 generateObject Pattern
```ts
import { generateObject } from 'ai';
import { anthropic } from '@ai-sdk/anthropic';
import { z } from 'zod';

const { object } = await generateObject({
  model: anthropic('claude-sonnet-4-20250514'),
  schema: z.object({ ... }),
  prompt: '...',
  maxTokens: 2000,
});
```

### Expo Router param passing for RecipeDetail
```ts
// Passing recipe via params:
router.push({
  pathname: '/(plugins)/pantry/recipe-detail',
  params: { recipe: JSON.stringify(recipe) },
});
// Receiving:
const { recipe: recipeStr } = useLocalSearchParams();
const recipe: Recipe = JSON.parse(recipeStr as string);
```

### HTTP call from PantryRecipes screen
```ts
const res = await fetch(`${EXPO_PUBLIC_API_URL}/pantry/recipes/suggest`, {
  method: 'POST',
  headers: { 'Authorization': `Bearer ${session.access_token}`, 'Content-Type': 'application/json' },
  body: JSON.stringify({ preferences }),
});
```

### Known Constraint
- STATE.md blocker: "Inject macro summary into system prompt via fetchUserContext to protect 5-step tool-call budget" — this Phase uses a DEDICATED endpoint (not the conversational agent), so there is NO tool-call budget concern here. The blocker applies to the conversational agent context only.

</code_context>

<specifics>
## Specific Ideas

- Macro budget banner uses a 2000 kcal default (universal baseline) — no need for user-specific calorie target which would require a new DB field
- Recipe generation takes ~5-10 seconds — show a skeleton loader / animated pulse on the cards
- "Regenerate" button on PantryRecipes screen to get fresh suggestions without changing preference
- RecipeDetail back button returns to PantryRecipes (no stack confusion since route params carry all data)

</specifics>
