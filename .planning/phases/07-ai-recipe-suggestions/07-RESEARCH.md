# Phase 7: AI Recipe Suggestions - Research

**Researched:** 2026-03-29
**Domain:** Vercel AI SDK v6 `generateObject`, Zod v4 schemas, Hono route mounting, Expo Router param passing
**Confidence:** HIGH

## Summary

Phase 7 adds AI-generated recipe suggestions to the pantry plugin. The implementation uses `generateObject` from the `ai` package (v6.0.116) with a Zod v4 schema to produce structured `Recipe[]` output from a dedicated `POST /pantry/recipes/suggest` endpoint. The frontend passes recipe data to a detail screen via JSON-encoded Expo Router params.

All three technical concerns — structured AI output, backend route mounting, and mobile param passing — are well-supported by the existing codebase patterns with minimal new dependencies. Zod v4.3.6 is already installed at the monorepo root (compatible with `ai` peer dep `^3.25.76 || ^4.1.8`). The `ai` package's `generateObject` function is confirmed exported and functional. The only new backend dependency is `zod` added explicitly to `backend/api/package.json`.

**Primary recommendation:** Use `generateObject` + Zod v4 `z.object` with `output: 'object'` (default) for the recipe endpoint. Mount the new router at `/pantry` in `app.ts`. Pass recipe JSON via `router.push` params and parse with `useLocalSearchParams`.

### Pattern 1: `generateObject` with Zod v4

**What:** Calls the AI model with a Zod schema. The SDK serializes the schema to JSON Schema, instructs the model to return conforming JSON, validates the output, and throws `NoObjectGeneratedError` on failure.

**When to use:** Any time structured, typed output is needed from the AI (not free-form text).

**Exact import and usage:**
```typescript
// Source: verified against node_modules/ai/dist/index.d.ts + node_modules/zod v4.3.6
import { generateObject } from 'ai';
import { anthropic } from '@ai-sdk/anthropic';
import { z } from 'zod';

const RecipeIngredientSchema = z.object({
  name: z.string().describe('Ingredient name'),
  quantity: z.number().describe('Amount'),
  unit: z.string().describe('Unit of measurement (g, ml, pieces, etc.)'),
});

const RecipeMacrosSchema = z.object({
  calories: z.number().describe('Total calories for base_servings'),
  protein_g: z.number().describe('Protein in grams'),
  carbs_g: z.number().describe('Carbohydrates in grams'),
  fat_g: z.number().describe('Fat in grams'),
});

const RecipeSchema = z.object({
  id: z.string().describe('Unique short slug, e.g. "poulet-roti"'),
  name: z.string().describe('Recipe name in French'),
  description: z.string().describe('One sentence description'),
  prep_time_min: z.number().describe('Preparation + cooking time in minutes'),
  base_servings: z.number().describe('Number of servings this recipe makes (1 or 2)'),
  ingredients: z.array(RecipeIngredientSchema),
  macros: RecipeMacrosSchema,
  steps: z.array(z.string()).describe('Cooking steps in order'),
});

const ResponseSchema = z.object({
  recipes: z.array(RecipeSchema),
});

// Usage in route handler:
const { object } = await generateObject({
  model: anthropic('claude-sonnet-4-20250514'),
  schema: ResponseSchema,
  maxTokens: 2000,
  system: 'You are a nutrition-aware chef assistant...',
  prompt: 'Generate 3 recipes using these pantry items...',
});

// object is typed as { recipes: Array<{ id: string, name: string, ... }> }
return c.json({ recipes: object.recipes, remaining_macros: remainingMacros });
```

**Return value:** `{ object, usage, finishReason, warnings, request, response }`. Only `object` is needed for the response.

**zod v4 import note:** In zod v4, the primary export is the `z` namespace (not a default export). Use:
```typescript
import { z } from 'zod';       // Named import — correct for zod v4
```
Not `import z from 'zod'` (though the default also works due to CJS interop).

### Pattern 3: Expo Router Param Passing for Complex Objects

**What:** JSON-stringify the recipe into a route param string. Parse on the detail screen with `useLocalSearchParams`.

**Limitation:** Expo Router encodes params as URL query strings. Large objects may hit URL length limits (~2000 chars in some environments). A typical Recipe with 3 steps and 6 ingredients is ~400-600 chars of JSON — well within safe limits.

**Send:**
```typescript
// Source: CONTEXT.md canonical pattern + Expo Router v4 docs pattern

const router = useRouter();

// On recipe card tap:

  pathname: '/(plugins)/pantry/recipe-detail',
  params: { recipe: JSON.stringify(recipe) },
});
```

**Receive:**
```typescript
import { useLocalSearchParams } from 'expo-router';
import type { Recipe } from '@ziko/plugin-pantry/types/recipe';
```

### Pattern 4: Supabase Data Fetch in Endpoint (self-contained)

The new endpoint fetches both pantry items and today's nutrition in parallel, reusing patterns from `tools/pantry.ts` and `context/user.ts`:

```typescript
import { clientForUser } from '../tools/db.js';

const today = new Date().toISOString().split('T')[0];
const db = clientForUser(userToken);

const [pantryRes, nutritionRes] = await Promise.all([
  db.from('pantry_items').select('name, quantity, unit, food_category, storage_location')
    .eq('user_id', userId).order('name'),
  db.from('nutrition_logs').select('calories, protein_g, carbs_g, fat_g')
    .eq('user_id', userId).eq('date', today),
]);
```

Compute `remaining_macros` by subtracting today's totals from the 2000/150/250/65 defaults (D-12).

### Anti-Patterns to Avoid

- **Using `jsonSchema()` from `ai` instead of Zod:** `jsonSchema()` works but loses TypeScript inference on `object`. Use Zod for typed output.
- **Importing `z` as default:** `import z from 'zod'` may cause issues in strict ESM contexts. Use named import `{ z }`.
- **Passing `recipe` param without JSON.stringify:** Expo Router params are strings; objects are silently coerced. Always stringify.
- **Using `router.replace` instead of `router.push`:** Use `push` so the back button returns to the recipes list.
- **Calling `generateObject` with `output: 'array'` + schema `z.array(...)`:** When using `output: 'array'`, pass the element schema (not wrapped in `z.array`). For this phase, use default `output: 'object'` with a wrapper `z.object({ recipes: z.array(RecipeSchema) })` — this is cleaner and matches D-03/D-05.

## Common Pitfalls

### Pitfall 1: Zod v4 Breaking Changes vs v3
**What goes wrong:** Code written for zod v3 may fail in v4. The most relevant breaking change: `z.object().shape` is no longer directly iterable in the same way. `z.describe()` is now `z.string().describe()` (method chaining) — this actually works the same.

**Why it happens:** Zod v4 was released in 2025 and has API differences from v3.

**How to avoid:** Use `z.object({ ... }).describe(...)` for schema-level descriptions. Use `z.string().describe(...)` for field-level. Both work in v4. Avoid `z.preprocess`, `z.transformer` (renamed in v4).

**Confidence:** MEDIUM (verified basic API works; full v4 changelog not exhaustively reviewed)

### Pitfall 2: `NoObjectGeneratedError` Not Caught
**What goes wrong:** If Claude fails to produce valid JSON matching the schema, `generateObject` throws `NoObjectGeneratedError`. Unhandled, this returns a 500 with a stack trace.

**Why it happens:** Model occasionally produces malformed JSON or misses required fields.

**How to avoid:** Wrap the `generateObject` call in a try/catch. Return `{ error: 'Failed to generate recipes' }` with status 500:
```typescript
import { NoObjectGeneratedError } from 'ai';
try {
  const { object } = await generateObject({ ... });
  return c.json({ recipes: object.recipes, remaining_macros });
} catch (err) {
  if (err instanceof NoObjectGeneratedError) {
    console.error('[Recipe Generation Error]', err.text);
    return c.json({ error: 'Recipe generation failed. Please try again.' }, 500);
  }
  throw err;
}
```

### Pitfall 3: URL Length Limit for Recipe Params
**What goes wrong:** Very large recipes (10+ steps, 15+ ingredients) could produce JSON strings exceeding URL param limits.

**Why it happens:** Expo Router encodes params into the URL.

**How to avoid:** The Zod schema limits each recipe to reasonable field sizes. `maxTokens: 2000` on the endpoint caps total output. A 3-recipe response with 6 ingredients and 5 steps each is ~800 chars JSON per recipe — safe. No mitigation needed for this phase, but be aware.

### Pitfall 4: `app.route('/pantry', ...)` Conflicts with Existing Route
**What goes wrong:** `app.route('/pantry', pantryRecipesRouter)` would conflict if a `/pantry` route already exists in `app.ts`.

**Why it happens:** Hono allows duplicate prefix mounts but the order matters.

**How to avoid:** Check `app.ts` — confirmed no existing `/pantry` route. The new mount is safe. The existing `supplements` and `bugs` routers use different prefixes.

### Pitfall 5: i18n Key Naming Conflicts
**What goes wrong:** Adding keys to `i18n.ts` with names that clash with existing `pantry.*` keys.

**Why it happens:** The existing pantry keys run from `pantry.title` to `pantry.camera_required_desc`. Recipe keys must start with `pantry.recipes.` or `pantry.recipe_detail.` as per D-17.

**How to avoid:** Add keys after the last `pantry.camera_required_desc` entry in both `fr` and `en` dictionaries. Use the `pantry.recipes.*` and `pantry.recipe_detail.*` namespace per D-17.

```typescript
const adjusted = {
```

### Remaining macro computation (backend)
```typescript
const DAILY_TARGETS = { calories: 2000, protein_g: 150, carbs_g: 250, fat_g: 65 };
const meals = nutritionRes.data ?? [];
const consumed = {
  calories: meals.reduce((s: number, m: any) => s + (Number(m.calories) || 0), 0),
  protein_g: meals.reduce((s: number, m: any) => s + (Number(m.protein_g) || 0), 0),
  carbs_g: meals.reduce((s: number, m: any) => s + (Number(m.carbs_g) || 0), 0),
  fat_g: meals.reduce((s: number, m: any) => s + (Number(m.fat_g) || 0), 0),
};
const remaining_macros = {
  calories: Math.max(0, DAILY_TARGETS.calories - consumed.calories),
  protein_g: Math.max(0, DAILY_TARGETS.protein_g - consumed.protein_g),
  carbs_g: Math.max(0, DAILY_TARGETS.carbs_g - consumed.carbs_g),
  fat_g: Math.max(0, DAILY_TARGETS.fat_g - consumed.fat_g),
};
```
