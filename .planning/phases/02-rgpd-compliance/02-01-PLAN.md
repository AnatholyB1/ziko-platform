1. Create `src/lib/ratelimit.ts` — Upstash sliding window singleton (per D-16, D-18):
   ```typescript
   import { Ratelimit } from '@upstash/ratelimit';
   import { Redis } from '@upstash/redis';

   export const ratelimit = new Ratelimit({
     redis: Redis.fromEnv(),
     limiter: Ratelimit.slidingWindow(5, '60 s'),
     prefix: 'ziko:delete',
   });
   ```
   `Redis.fromEnv()` reads `UPSTASH_REDIS_REST_URL` and `UPSTASH_REDIS_REST_TOKEN` automatically.
   Do NOT add `analytics: true` (free tier issue per research).

2. Create `src/actions/account.ts` with `'use server'` directive at the top.

   Export the state type:
   ```typescript
   export type DeleteAccountState = {
     status: 'idle' | 'success' | 'error';
     message: string;
   };
   ```

   Export the `deleteAccount` function with signature `(prevState: DeleteAccountState, formData: FormData) => Promise<DeleteAccountState>`.

   Implementation steps inside the function:
   a. Extract IP: `const headerList = await headers();` (import `headers` from `'next/headers'`). Parse IP from `x-forwarded-for`, split on comma, trim, fallback to `'127.0.0.1'` (per D-17).
   b. Rate limit: `const { success } = await ratelimit.limit(ip);` — if `!success`, return `{ status: 'error', message: 'Trop de tentatives. Reessayez dans une minute.' }`.
   c. Validate form: extract `email` (trimmed, lowercased) and `confirmation` from formData. If `!email` or `confirmation !== 'SUPPRIMER'`, return `{ status: 'error', message: 'Formulaire invalide.' }`.
   d. Find user by email — create a private `findUserIdByEmail` function that uses `fetch()` against `${process.env.SUPABASE_URL}/auth/v1/admin/users?filter=${encodeURIComponent(email)}` with headers `{ apikey: process.env.SUPABASE_SERVICE_ROLE_KEY!, Authorization: 'Bearer ' + process.env.SUPABASE_SERVICE_ROLE_KEY! }`. Parse the JSON response, find user where `u.email?.toLowerCase() === email`. Return `id` or `null`.

      CRITICAL: Do NOT use `auth.admin.getUserByEmail()` — it does not exist in supabase-js v2.100.1. The research confirmed this from source code inspection.

      If the REST `?filter=` approach fails (non-200 response), fall back to: `const { data } = await admin.auth.admin.listUsers({ page: 1, perPage: 1000 }); return data?.users.find(u => u.email === email)?.id ?? null;`

   e. Anti-enumeration (per Pitfall 6 in research): If `userId` is null (email not found), return `{ status: 'success', message: 'Si ce compte existe, il a ete supprime.' }` — same success message as a real deletion.
   f. Delete: `const { error } = await admin.auth.admin.deleteUser(userId);` — if error, return `{ status: 'error', message: 'Erreur lors de la suppression. Contactez le support.' }`.
   g. On success: return `{ status: 'success', message: 'Votre compte a ete supprime definitivement.' }`.

   Use proper French characters (accents) in all user-facing strings: e with accent = \u00e9, a with accent = \u00e0, etc. Write them as literal characters, not escape sequences.
