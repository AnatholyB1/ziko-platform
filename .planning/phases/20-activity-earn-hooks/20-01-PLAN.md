**1. Add POST /earn to credits.ts (per D-06, D-07):**

After the existing `router.get('/balance', ...)` block and before the `export`, add:

```typescript
// EARN-01..06: POST /credits/earn — mobile-side earn path (D-06)
router.post('/earn', async (c) => {
  const { userId } = c.get('auth');
  const body = await c.req.json().catch(() => ({}));
  const { source, idempotency_key } = body as { source?: string; idempotency_key?: string };

  if (!source || !idempotency_key) {
    return c.json({ credited: false }, 200); // Per D-06: always 200, silent failure
  }

  const VALID_SOURCES = ['workout', 'habit', 'meal', 'measurement', 'stretch', 'cardio'];
  if (!VALID_SOURCES.includes(source)) {
    return c.json({ credited: false }, 200);
  }

  const result = await creditService.earnCredits(userId, source, idempotency_key);
  return c.json({ credited: result.credited });
});
```

Response is ALWAYS 200 with `{ credited: boolean }`. Validation failures return `{ credited: false }`. Auth is already enforced by `router.use('*', authMiddleware)` at the top.

<threat_model>
## Trust Boundaries

| Boundary | Description |
|----------|-------------|
| client -> POST /credits/earn | Untrusted mobile client sends source + idempotency_key |
| tool executor -> creditService | Server-side trusted — userId comes from auth middleware |

## STRIDE Threat Register

| Threat ID | Category | Component | Disposition | Mitigation Plan |
|-----------|----------|-----------|-------------|-----------------|
| T-20-01 | Spoofing | POST /credits/earn | mitigate | authMiddleware validates Supabase JWT before handler runs; userId extracted from verified token, not request body |
| T-20-02 | Tampering | POST /credits/earn body.source | mitigate | Validate source against VALID_SOURCES allowlist ['workout','habit','meal','measurement','stretch','cardio']; reject unknown sources with credited:false |
| T-20-03 | Elevation of Privilege | POST /credits/earn unlimited calls | mitigate | creditService.earnCredits enforces DAILY_EARN_CAP=4; idempotency key prevents double-credit; existing rate limiter from Phase 12 applies |
| T-20-04 | Denial of Service | earnCredits failure in tool executor | mitigate | Fire-and-forget pattern (.catch(() => {})) ensures earn failure never blocks the activity save (SC-3) |
| T-20-05 | Repudiation | earn transaction without audit | accept | ai_credit_transactions table logs every earn with user_id, source, idempotency_key, created_at — sufficient audit trail |
</threat_model>
